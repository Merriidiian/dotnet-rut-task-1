# Задание: Коллекции и ООП — консольное приложение по слоям

## Цель

Реализовать **одно консольное приложение** с разделением на **три слоя** и связью между ними по принципу **D (SOLID)** — зависимость от абстракций (интерфейсов), а не от конкретных классов.

В приложении должны быть явно видны:
- **«База данных»** — хранение в коллекциях в памяти (лекции: **collection/02_List**, **collection/03_Dictionary**).
- **Запросы** — выборки и отчёты через LINQ (лекции: **collection/06_LINQ_Basics**, **collection/07_LINQ_More_Operators**, **collection/08_LINQ_Grouping_And_Aggregation**).
- **ООП** — классы, инкапсуляция, композиция, интерфейсы (лекции: **oop/01_Classes_And_Objects**, **oop/02_Encapsulation**, **oop/12_Composition_And_More**, **oop/11_SOLID_Dependency_Inversion**).

---

## Три слоя

### 1. Консольный интерфейс (UI)

- **Бесконечное** консольное приложение: цикл меню, пользователь выбирает пункт, приложение выполняет действие и снова показывает меню.
- Консоль **не знает** о репозиториях и базе — только о **сервисах** (через интерфейсы `IOrderService`, `ICatalogService`, `IExpenseService` и т.д.).
- Ссылки на темы: **oop/01_Classes_And_Objects** (точка входа, вызов методов объектов).

### 2. Сервисный слой

- **Простая логика**: создание заказа, добавление товара, подсчёт выручки, отчёты по категориям и т.д.
- Сервисы зависят только от **интерфейсов репозиториев** (`IOrderRepository`, `IProductRepository` и т.д.), а не от конкретных классов — это и есть **D (SOLID)** (лекция **oop/11_SOLID_Dependency_Inversion**).
- Запросы к данным оформляются через **LINQ** в сервисах: `Where`, `GroupBy`, `Sum`, `OrderBy`, `Take` (лекции **collection/06**, **07**, **08**).
- Один сервис — одна зона ответственности (лекция **oop/07_SOLID_Single_Responsibility**).

### 3. Репозиторий + база данных

- **Репозиторий** — класс, который хранит данные в коллекциях (`List<T>`, `Dictionary<TKey, TValue>`) и даёт к ним доступ (добавить, получить по id, получить все). Коллекции объявлены **явно** в коде — это и есть «видимая база» (лекции **collection/02_List**, **collection/03_Dictionary**).
- Репозиторий реализует **интерфейс** (например, `IOrderRepository`). Конкретный класс с `List<Order> _orders` создаётся в точке входа (Program) и передаётся в сервис как `IOrderRepository` — инверсия зависимостей.
- Без реальной СУБД: только память (лекция **collection/01_Arrays_And_Intro** — идея хранения набора данных; здесь — динамические списки и словари).

---

## Связь между слоями (D — SOLID)

- **Консоль** получает в конструкторе (или методе) **интерфейсы сервисов**: `IOrderService`, `ICatalogService`, `IExpenseService`. Не создаёт сервисы сама (`new OrderService(...)` только в Program).
- **Сервисы** получают в конструкторе **интерфейсы репозиториев**: `IOrderRepository`, `ICustomerRepository`, `IProductRepository` и т.д. Не создают репозитории сами.
- **Program** (или конфигурация) создаёт конкретные репозитории и сервисы и «собирает» приложение: `new OrderRepository()` → `new OrderService(orderRepo, customerRepo)` → `new AppMenu(orderService, ...)`.

Так верхние слои не зависят от нижних напрямую — только от абстракций (лекция **oop/11_SOLID_Dependency_Inversion**).

## Предметные области (минимум 2–3)

Одно приложение — несколько доменов. Примеры:

| Область        | Сущности                          | Примеры запросов (LINQ в сервисе)                    |
|----------------|------------------------------------|------------------------------------------------------|
| **Заказы**     | Клиент, Заказ, Позиция, enum статус | Заказы по статусу/клиенту, сумма по клиентам (GroupBy) |
| **Каталог**    | Категория, Товар                   | Поиск по имени (Where), топ по цене (OrderBy + Take), статистика по категориям (GroupBy, Sum) |
| **Расходы**    | Категория расхода, Транзакция      | Сумма за период (Where + Sum), по категориям (GroupBy), последние N (OrderBy + Take) |

## Требования к коду

- **ООП**: классы для сущностей; инкапсуляция (приватные поля, доступ через свойства/методы — **oop/02_Encapsulation**); композиция (заказ содержит список позиций — **oop/12_Composition_And_More**); enum для статусов.
- **Коллекции**: явные `List<>`, `Dictionary<>` в репозиториях; LINQ в сервисах.
- **Интерфейсы**: для каждого репозитория и сервиса, используемого снаружи — чтобы консоль зависела от `IOrderService`, сервис от `IOrderRepository` и т.д.
- **Один класс — один файл.** Логичная раскладка по папкам: `Entities/`, `Repositories/`, `Services/`, `Console/` (или `UI/`).
---

## Критерии оценки

- Реализованы **три слоя**: консоль (бесконечное меню), сервисы, репозитории + «база» (коллекции).
- Между слоями — **D (SOLID)**: консоль от интерфейсов сервисов, сервисы от интерфейсов репозиториев.
- Минимум **2–3 предметные области**, запросы через **LINQ**, данные в **List/Dictionary** в репозиториях.
- Каждый класс в отдельном файле, папки по слоям/доменам.

├── Entities/                       # Один класс — один файл (ООП: oop/01, 02, 12)
│   ├── Customer.cs
│   ├── Order.cs
│   ├── OrderItem.cs
│   ├── OrderStatus.cs
│   ├── Category.cs
│   ├── Product.cs
│   ├── ExpenseCategory.cs
│   └── Transaction.cs
├── Repositories/                   # «База» (List, Dictionary) + интерфейсы (D-SOLID)
│   ├── ICustomerRepository.cs
│   ├── CustomerRepository.cs
│   ├── IOrderRepository.cs
│   ├── OrderRepository.cs
│   ├── ICategoryRepository.cs
│   ├── CategoryRepository.cs
│   ├── IProductRepository.cs
│   ├── ProductRepository.cs
│   ├── IExpenseCategoryRepository.cs
│   ├── ExpenseCategoryRepository.cs
│   ├── ITransactionRepository.cs
│   └── TransactionRepository.cs
├── Services/                       # Логика + LINQ, зависимость от интерфейсов репозиториев
│   ├── IOrderService.cs
│   ├── OrderService.cs
│   ├── ICatalogService.cs
│   ├── CatalogService.cs
│   ├── IExpenseService.cs
│   └── ExpenseService.cs
